# 선호

- 유닛 테스트보다 실제 사용 요구, 흐름을 검증하는 인수, 통합 테스트 선호

# 바퀴 만들기 및 실환경성

- 테스트 라이브러리, 유틸 등 기존 테스트 더블, 인프라 적극 재사용
- 개별 테스트에 흩어진 mock, stub, spy 대신 setupIntegration 같은 이름의 공통
  페이크(fake) 모듈로 통합/사용
- 페이크는 원본과 최대한 동일한 인터페이스 유지
  - 다른 저장소 재사용 가능성을 고려해 페이크 설계
  - 사용처에서 예상 가능한 페이크 구현 - vi.fn() 같은 유틸 활용 순 선호
  - 시나리오별 조건, 의도는 테스트 안에서 드러나도록 KISS 염두
- 테스트별 관심사와 무관한 외부 경계 차이를 최대한 공통 하니스에 흡수해 각
  테스트가 동일한 실행 환경 위에서 시나리오만 바꾸는 형태로 사용

- 실환경성: 실제 런타임, 외부 의존성, 배포 산출물과의 거리
- 적절한 페이크 인프라가 없다면 외부 시스템 조합, 스키마, 런타임 차이, 위험도를
  보고 먼저 도입하거나 유저에게 제안

- 실환경성이 충분히 높은 테스트가 전혀 없다면, 핵심 경로 하나만 확인하는 스모크
  테스트라도 추가 시도

# 격리 규칙

- **외부 경계에만** 페이크 사용. 외부 경계란
  - 프로세스 밖 I/O: 네트워크, 파일 시스템, 시간/타이머 등 비결정적 런타임 경계
  - 외부 소유 API/SDK: 브라우저·OS API, 네이티브 브릿지, 위 I/O API 등

내부 클래스/인터페이스, 훅·함수 옵션, 생성자 주입, Context Provider는 외부
경계가 **아닙니다**. 실제 조립 경로를 따라 외부 I/O까지 내려가 대체 지점을
잡습니다. 예: fetch를 쓰는 모듈 대신 fetch를 대체하거나 msw를 사용합니다.

페이크만을 위해 새 앱 내부 경계를 만들지 않습니다. 대체 지점은 프로덕션 런타임이
실제로 만나는 외부 소유 API입니다.

내부 모듈 연결은 프로덕션 조립 그대로 유지하고, 테스트 전용 하니스는 환경
준비·구동·관찰에 한정합니다. 결과적으로 대체가 외부 경계 한 층에만 생기고 원하는
부수 효과가 바뀌지 않는 한 내부 리팩토링이 테스트를 깨지 않아야 합니다.

페이크 작성/사용 전 자문:

1. 페이크 대체 대상이 외부 소유 API인가? 프로세스 밖 I/O, 플랫폼/브라우저/OS
   API, 또는 그 공식 SDK 객체인가?
   - 아니라면 실제 외부 경계까지 다시 추적합니다
2. 새로 만든 앱 내부 seam이 아니라, 작업 전 프로덕션 런타임이 이미 의존하던 외부
   API 호출 지점에 의존하는가?
   - 앱 내부 seam이면 리팩토링이며 레이어 추가입니다. 올바른 경계를 확인해 통합
     테스트로 작성하거나 이미 해당 유닛을 잘 커버하는 통합 테스트가 있다면 유닛
     테스트 작성을 허용합니다.
3. 새 앱 내부 경계 없이 해당 외부 경계를 페이크로 대체하거나 실제 구성 요소로
   구동할 수 있는가?
   - 곤란하다면 이유를 먼저 유저에게 알리고 제안합니다.

# 유닛 테스트

- 유닛/통합 구분은 테스트 대상 범위와 프로덕션 조립 경로를 검증하는지 여부로
  판단
- 함수의 입력/출력 검증 케이스를 늘리는 데 집중
- 클래스 테스트 셋업이 내부 상태, 콜백, 시간 제어, 비동기 대기 등으로 복잡해질
  수 있어
  - 가능한 함수를 선호하고 클래스 코드는 최소화해 조립 경로만 검증
  - 클래스 테스트는 내부 함수별 세부 규칙보다 공개 계약과 함수 연결 결과만 소수
    검증

# 고립

- 테스트 작성 시 한 프로세스 내 병렬 실행 고려
  - cwd 같은 전역 상태에 의존하지 않도록 주의

# Red-Green-Refactor

일회성, 생성 코드, 설정이 아닌 프로덕션 코드의 사용자 관찰 가능 동작, 공개 계약,
부수 효과를 추가하는 경우

- Red(test): 좋은 테스트를 만들어 의도한 이유로 실패함을 확인하고
- Green(feat): 가장 단순하게 테스트를 통과시킨 다음
- Refactor(refactor): 통과를 유지하면서 가독성을 위해 리팩토링합니다.
- 다른 Red를 추가해 반복합니다.

좋은 테스트: 하나만 테스트하고, 이름에서 의도가 드러나는 테스트

기능 제거처럼 기존 테스트가 사용자 요구를 충분히 보장하면 추가보다 수정을
선호합니다. 기능 부재 자체를 새 테스트로 고정하지 않습니다.

비기능 요구를 위한 코드라면 테스트를 추가하거나 코드를 제거합니다.

리팩토링 대상 실행 경로를 보장하는 상위 범위 테스트가 없다면 먼저 현재 동작을
고정하는 테스트를 추가한 뒤 대상 코드와 테스트를 수정합니다.

Red 테스트를 짜지 않기로 했다면 유저에게 불가피한 이유를 먼저 알립니다.
