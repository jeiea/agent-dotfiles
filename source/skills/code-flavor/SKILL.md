# 코드 작성 시

- 업계 표준, Best Practice에 부합할지 고려
- 테스트 계약을 준수하는 내에서 최소 수정이 아닌 최소 코드 지향
  - 코드 중복을 줄이기 위해 넓은 범위 수정이 필요한 경우 일회성 해결 대신 제안
- 가변 상태 최소화
- 코드 매 일부분에서 입력, 결과를 명확하게 최소로 유지
- 저장소 규칙, 언어 제약, 컨벤션 내에서
  - 임포트 부수효과, 공개 인터페이스, 공개 항목, 구현 상세 순으로 배치 (점강식)
  - 참조 코드가 피참조 코드보다 앞서도록 배치 (top-down)
  - 가령 TS에선 필수 const → 탑레벨 실행문 → export 항목 → 줄기 함수 → leaf 함수
    순으로 배치
- 코드 중복 서술 주석 대신 함수 추출 선호
- 배경, 근거, 참고 자료, 데이터 보존 염두
  - 배경: 유저 의도, 이전 방식의 문제, 시행착오, 폐기한 대안과 이유, 다른
    스쿼시된 커밋 요약, 이 커밋하고만 연관된 유저 프롬프트 등
  - 근거, 참고자료: 피그마, 슬랙, 이슈 URL 등
  - 데이터: 원시 데이터, 오류 메시지 등
  - 다음 중 한 군데 이상 삽입 고려
    - 코드, 주석: 코드의 의문을 해소해야 하는 경우
    - 문서: 기존 유관 부분이 있을 경우
    - 테스트 케이스 이름: 명확한 계약인 경우
    - 커밋 메시지: 변경 자체 또는 역사와 관련한 경우
  - 민감 정보는 제거 또는 익명화
    - 로컬 경로는 재현성도 없으므로 적절히 수정
- 데이터 흐름, 비즈니스 규칙(제약사항), 암묵지(바로 파악이 힘든 용어)를 가능한
  파악하고 의외이거나 작업 단축에 중요한 발견이 문서화되지 않았다면 문서화 제안
- 디노 컨벤션: 함수 인터페이스 설계 시 가능하다면 0-2개의 필수 인자와 필요 시
  마지막에 선택적 매개변수를 포함 가능한 plain object 내지 named parameter를
  받게 합니다

# 계획 시

- UI/UX, 보안, 성능, 오류 처리, 국제화, 확장성, 상호 운용성, 자원 관리, 장애
  허용, 테스트 전략, 변경 전략, 의존성/직접 구현 범위 기준, 재사용 결정 등 고려
- 각 변경이 꼭 필요한 추가인지, 생략 또는 더 간단한 대안 고려
- 원인 불명 시 추측성 픽스 금지. 우회/완화는 한계, 제거 조건, 진단 수단 명시
